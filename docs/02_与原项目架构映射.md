# 与原项目架构映射

## 1. 对照表

| 原工程 | 本示例 | 作用 |
|---|---|---|
| `IAAirBinder.aidl` | `IServiceRouter.aidl` | 一级 Binder 总路由 |
| `ConnectValue.SERVICE_NAME_*` | `ServiceNames.LIGHT/TIMER` | 二级服务路由键 |
| `AppAdapterService.getBinder()` | `RoomCoreService.routerBinder.getService()` | 服务端分发业务 Binder |
| `ConnectManager` / `AppConnectProxy` | `RoomConnection` | 绑定状态、超时、重连、主动断开 |
| `HvacProxy`、`TrackingProxy` | `LightProxy`、`TimerProxy` | 本地代理封装远程调用 |
| 各业务 `IA*Client.aidl` | `ILightService`、`ITimerService` | 领域远程合同 |
| `IAClientListener` 等回调 | `ILightCallback`、`ITimerCallback` | 服务端反向通知客户端 |
| `UIAdapterManager` / ViewManager | `MainActivity` | 消费代理回调并更新 UI |

## 2. 保留了哪些原理

### 一级 Service，多领域接口

本示例只有一个 Android Service，但它承载灯光与计时两个领域。扩展领域时只增加二级 Binder，不增加顶层 Service。

### 接口与实现分离

AIDL 只定义“能做什么”；`RoomCoreService` 决定“怎样做”；Proxy 决定“客户端怎样用”。

### 代理模式

UI 调 `LightProxy`，实际工作由远端 `LightBinder` 完成。调用者看起来像在使用本地对象。

### 模板方法

`BaseFeatureProxy` 固定生命周期，子类只实现：

```text
asInterface
registerRemoteCallback
unregisterRemoteCallback
```

### 单例

`RoomConnection`、`LightProxy`、`TimerProxy` 都用静态内部类实现懒加载单例，避免重复连接和重复领域状态。

### 同步与异步分工

同步 `int` 返回是否受理；Callback 返回执行后的状态。示例故意让灯光操作延迟 250ms，便于在日志里看清两者不是一回事。

## 3. 为教学做了哪些简化

| 原工程复杂点 | 本示例简化 |
|---|---|
| 多个独立 APK、平台签名与 exported Service | 一个 APK 的两个私有进程 |
| 数十个领域、四音区、车型差异 | 两个领域、单一房间 |
| 前台 Service 与常驻进程 | Activity 生命周期内的绑定 Service |
| 私有 Maven、AAR/JAR/SO、车身 API | 只使用 Android SDK |
| 大量 JSON 扩展协议 | 强类型 AIDL 参数 |
| 多套连接实现 | 一个共享 `RoomConnection` |

这些简化没有改变 Binder 机制本身。跨进程序列化、线程切换、死亡代理和反向回调仍是真实发生的。

## 4. 原工程阅读结论如何影响本示例

原工程中最容易让初学者混淆的是：绑定成功不等于每个二级 Binder 都存在；同步返回成功也不等于硬件已完成；Binder 回调也不等于主线程。

因此本示例把三种状态分开显示：

1. 顶层连接状态；
2. LIGHT/TIMER 二级 Binder 可用状态；
3. 领域最终业务状态。

这是理解大型项目时非常实用的排错分层。
