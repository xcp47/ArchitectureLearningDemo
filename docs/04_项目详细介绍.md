# ArchitectureLearningDemo 项目详细介绍

## 1. 项目定位

`ArchitectureLearningDemo` 是一个可以直接使用 Android Studio 打开的“小型智能自习室”项目。它的重点不是灯光和倒计时功能本身，而是用尽量少的代码复现大型车机工程中的核心通信架构：

```text
一个 APK、两个进程、两级 Binder、多个业务领域、统一连接、代理封装和异步回调
```

项目将原工程中的多 APK、平台签名、车机私有 SDK、车型差异和大量业务领域进行简化，但保留了真实的 Binder IPC、远端回调、线程切换、连接重试和进程异常处理。

## 2. 功能概览

应用提供控制端和被控端两类页面：

1. **连接管理**
   - 连接或主动断开远端 Service；
   - 显示一级 Binder 的连接状态；
   - 显示 UI 进程 PID 和远端进程 PID。
2. **灯光控制**
   - 开灯、关灯；
   - 调整 0～100% 亮度；
   - 服务端延迟 250ms，模拟真实硬件执行。
3. **专注计时**
   - 启动 10 秒或 30 秒倒计时；
   - 取消倒计时；
   - 服务端每秒回调剩余时间和总时间。
4. **调用日志**
   - 显示同步返回值；
   - 显示异步业务结果；
   - 显示回调来自哪个进程。
5. **被控端模拟屏**
   - 页面运行在 `:room_core` 进程；
   - 根据最终灯光回调切换明亮/暗黑模式；
   - 使用方框式 `XX / XX` 时钟显示剩余时间和总时间。

控制端布局见 [`activity_main.xml`](../app/src/main/res/layout/activity_main.xml)，被控端布局见 [`activity_device_preview.xml`](../app/src/main/res/layout/activity_device_preview.xml)。

## 3. 总体架构

```text
┌──────────────────── UI 主进程 ────────────────────┐
│                                                   │
│  MainActivity                                     │
│      │                                            │
│      ├── RoomConnection ── 管理一级 Binder 连接   │
│      ├── LightProxy ────── 灯光本地代理            │
│      └── TimerProxy ────── 计时器本地代理          │
│              │                                    │
└──────────────┼────────────────────────────────────┘
               │ AIDL / Binder IPC
               ▼
┌────────────── :room_core 远端进程 ────────────────┐
│                                                   │
│  RoomCoreService                                  │
│      │                                            │
│      └── IServiceRouter 一级 Binder               │
│              │                                    │
│              ├── "LIGHT" → ILightService          │
│              └── "TIMER" → ITimerService          │
│                                                   │
│  完成操作后，通过 Callback 反向通知 UI 进程        │
└───────────────────────────────────────────────────┘
```

项目只有一个 Android Service，但该 Service 可以承载多个业务 Binder。以后增加窗帘、音乐或空调领域时，可以增加新的二级 AIDL 和 Binder 实现，而不必为每个领域发布一个顶层 Service。

## 4. 两个真实进程

[`AndroidManifest.xml`](../app/src/main/AndroidManifest.xml) 将 `RoomCoreService` 配置到私有远端进程：

```xml
<service
    android:name=".service.RoomCoreService"
    android:exported="false"
    android:process=":room_core" />
```

`MainActivity` 位于应用主进程，`RoomCoreService` 位于 `:room_core` 进程。因此这里会真实发生：

- Binder 跨进程通信；
- 参数序列化和反序列化；
- Binder 线程调度；
- `RemoteException`；
- 远端进程断开；
- 反向 Binder Callback。

把两端放在同一个 APK 中只是为了方便安装和学习，不会把 IPC 变成普通的本地 Java 调用。

## 5. 两级 Binder

### 5.1 一级 Binder：服务路由器

[`IServiceRouter.aidl`](../app/src/main/aidl/com/example/smartroom/ipc/IServiceRouter.aidl) 只定义一个路由方法：

```java
IBinder getService(String serviceName);
```

客户端绑定 `RoomCoreService` 后，先获得 `IServiceRouter`，再通过统一服务名请求业务 Binder：

```java
router.getService(ServiceNames.LIGHT);
router.getService(ServiceNames.TIMER);
```

一级 Binder 负责“找到哪个领域”，不执行具体业务。

### 5.2 二级 Binder：领域服务

灯光领域由 [`ILightService.aidl`](../app/src/main/aidl/com/example/smartroom/ipc/ILightService.aidl) 定义：

```java
interface ILightService {
    void registerCallback(ILightCallback callback);
    void unregisterCallback(ILightCallback callback);
    int setEnabled(boolean enabled);
    int setBrightness(int brightness);
}
```

计时领域由 [`ITimerService.aidl`](../app/src/main/aidl/com/example/smartroom/ipc/ITimerService.aidl) 定义：

```java
interface ITimerService {
    void registerCallback(ITimerCallback callback);
    void unregisterCallback(ITimerCallback callback);
    int start(int seconds);
    int cancel();
}
```

二级 Binder 负责“这个领域具体能做什么”。

## 6. 一次开灯调用的完整过程

```text
用户点击“开 / 关灯”
        │
        ▼
MainActivity 调用 LightProxy.setEnabled(true)
        │
        ▼
LightProxy 调用远端 ILightService.setEnabled(true)
        │ Binder IPC
        ▼
RoomCoreService.LightBinder 收到请求
        │
        ├── 立即返回 0：请求已受理
        │
        └── 250ms 后修改灯光状态
                    │
                    ▼
          ILightCallback.onLightChanged(...)
                    │ 反向 Binder IPC
                    ▼
          LightProxy 切换到 UI 主线程
                    │
                    ▼
          MainActivity 更新页面和日志
```

日志会先出现同步返回：

```text
同步返回 ← LightProxy.setEnabled(true)：0（请求已受理）
```

约 250ms 后再出现异步结果：

```text
异步回调 → onLightChanged(true, 50)，来自 PID=1260
```

这里最重要的结论是：

```text
同步返回成功 ≠ 硬件已经完成
```

同步返回值只表示服务端是否受理请求，最终业务状态由 Callback 返回。

## 7. 统一连接层 RoomConnection

[`RoomConnection.java`](../app/src/main/java/com/example/smartroom/client/RoomConnection.java) 只负责一级 Router Binder，不认识灯光或计时器业务。

```text
DISCONNECTED
     │ start()
     ▼
CONNECTING
     │ onServiceConnected()
     ▼
CONNECTED
     │ 远端断开或绑定失效
     ▼
DISCONNECTED → 自动重试
```

连接策略包括：

- 连接超时为 2 秒；
- 前五次以 500ms 间隔快速重试；
- 后续以 3 秒间隔慢速重试；
- 处理 `onServiceDisconnected()`；
- 处理 `onBindingDied()`；
- 处理 `onNullBinding()`；
- 业务 Proxy 捕获 `RemoteException` 后请求重新连接；
- `stop()` 表示主动断开，会停止自动重连。

区分主动断开和异常断开非常重要，否则用户点击“断开”之后，连接状态机可能立即把服务重新连接起来。

## 8. 代理层

`MainActivity` 不直接接触 `IBinder`、`ServiceConnection`、AIDL Stub 或 `RemoteException`，而是调用普通 Java 风格的方法：

```java
lightProxy.setEnabled(true);
timerProxy.startTimer(10);
```

[`LightProxy.java`](../app/src/main/java/com/example/smartroom/client/LightProxy.java) 和 [`TimerProxy.java`](../app/src/main/java/com/example/smartroom/client/TimerProxy.java) 负责：

- 保存强类型远程 AIDL 接口；
- 判断二级 Binder 是否可用；
- 注册和注销远程 Callback；
- 捕获 `RemoteException`；
- 返回统一结果码；
- 将 Binder 回调切换到 UI 主线程；
- 将远程状态转换成本地 Listener 通知。

这样，UI 层只表达业务意图，不承担 IPC 基础设施代码。

## 9. BaseFeatureProxy 模板方法

灯光和计时代理都必须完成以下流程：

```text
取得二级 Binder
→ 转成强类型 AIDL 接口
→ 注册 Callback
→ 远端断开时清理
→ 停止时注销 Callback
```

[`BaseFeatureProxy.java`](../app/src/main/java/com/example/smartroom/client/BaseFeatureProxy.java) 固定公共骨架，子类只实现领域差异：

```java
asInterface(...)
registerRemoteCallback(...)
unregisterRemoteCallback(...)
onAvailabilityChanged(...)
```

新增窗帘或音乐代理时，可以复用同一套生命周期和异常处理逻辑。

## 10. 服务端实现

[`RoomCoreService.java`](../app/src/main/java/com/example/smartroom/service/RoomCoreService.java) 内部包含：

```text
routerBinder
├── LightBinder
└── TimerBinder
```

### LightBinder

灯光服务保存开关状态和亮度。Binder 方法只做参数校验和任务排队，通过 `Handler.postDelayed()` 模拟硬件在 250ms 后完成操作，然后广播最终状态。

### TimerBinder

计时服务保存剩余秒数和运行状态，使用 `Runnable ticker` 每秒递减，并向客户端广播最新结果。

### RemoteCallbackList

服务端使用 `RemoteCallbackList` 管理跨进程 Callback，而不是普通的 Java `List`。它更适合处理客户端 Binder 死亡和失效回调。

广播时需要成对调用：

```java
int count = callbacks.beginBroadcast();
try {
    // 遍历通知客户端
} finally {
    callbacks.finishBroadcast();
}
```

## 11. 线程模型

```text
UI 主线程
   │
   ▼
远端 Binder 线程
   │ Handler.post()
   ▼
远端 Service 主线程
   │ Callback IPC
   ▼
客户端 Binder 线程
   │ mainHandler.post()
   ▼
UI 主线程
```

Callback AIDL 使用 `oneway`，服务端发出通知时不需要等待客户端处理完成。Callback 到达客户端时不保证位于 UI 主线程，因此 Proxy 必须用主线程 `Handler` 再分发，之后 `MainActivity` 才能安全更新 View。

## 12. Activity 生命周期

[`MainActivity.java`](../app/src/main/java/com/example/smartroom/MainActivity.java) 在 `onStart()` 中启动代理并连接服务：

```java
lightProxy.start();
timerProxy.start();
connection.start(this);
```

在 `onStop()` 中注销监听并主动断开：

```java
lightProxy.stop();
timerProxy.stop();
connection.stop();
```

因此当前远端连接跟随页面生命周期，页面离开后不会继续长期保持连接。这适合教学项目；车机常驻服务通常会使用更长的应用级生命周期。

## 13. 设计模式

| 模式 | 对应实现 | 作用 |
|---|---|---|
| Router / Facade | `IServiceRouter` | 用一个入口获取多个业务 Binder |
| Proxy | `LightProxy`、`TimerProxy` | 隐藏 IPC 和远程异常 |
| Template Method | `BaseFeatureProxy` | 复用业务代理生命周期 |
| Singleton | `RoomConnection` 和两个 Proxy | 避免重复绑定和重复状态 |
| Observer | Listener、AIDL Callback | 异步分发状态变化 |
| State Machine | `RoomConnection` | 管理连接、超时、断开和重试 |
| Service Registry | `ServiceNames` | 通过名称查找领域服务 |

## 14. 与原车机项目的映射

| 原工程概念 | 教学项目 |
|---|---|
| `IAAirBinder` | `IServiceRouter` |
| `AppAdapterService.getBinder()` | `RoomCoreService.getService()` |
| `ConnectManager` | `RoomConnection` |
| `HvacProxy` 等领域代理 | `LightProxy`、`TimerProxy` |
| 业务 Client AIDL | `ILightService`、`ITimerService` |
| 业务 Listener AIDL | `ILightCallback`、`ITimerCallback` |
| UIAdapter / ViewManager | `MainActivity` |

原工程拥有多个 APK、几十个领域、系统权限、车型配置和真实硬件接口。本项目保留通信骨架，去掉不影响架构理解的依赖。

## 15. 当前简化和边界

教学版本暂未实现：

- 数据库和数据持久化；
- 依赖注入；
- 自动化测试；
- 前台常驻 Service；
- 多客户端权限校验；
- 真实硬件控制；
- `Parcelable` 复杂对象；
- 手动 `linkToDeath` 死亡监听；
- 多模块或多 APK 拆分。

灯光和计时状态保存在远端进程内存中，远端进程被杀后状态会重置。这也说明进程内存不能替代持久化存储。

## 17. 被控端模拟屏

`DevicePreviewActivity` 与 `RoomCoreService` 都运行在 `:room_core`。控制端通过“查看被控端效果”打开该页面后，可以观察：

- 明亮模式：暖色房间面板和点亮状态；
- 暗黑模式：深色房间面板和熄灭状态；
- 服务端确认的亮度百分比；
- `XX / XX` 格式的方框倒计时；
- 被控端 Activity 与 Service 的相同 PID。

计时 Callback 现在同时携带 `remainingSeconds` 和 `totalSeconds`，所以中途进入页面也能得到正确分母。该页面不提供控制按钮，只展示 Binder 服务端完成后的最终状态。

单 APK 双进程和两个独立 APK 的系统性比较见 [06_单APK双进程与双APK对比.md](06_单APK双进程与双APK对比.md)。
