# 控制端与被控端说明

## 1. 当前程序如何划分两端

当前项目同时包含控制端和被控端，两端位于同一个 APK 的两个独立进程：

~~~text
控制端（应用主进程）
MainActivity
  └─ LightProxy / TimerProxy
          │ AIDL / Binder IPC
          ▼
被控端（:room_core 进程）
RoomCoreService
  ├─ LightBinder
  ├─ TimerBinder
  └─ DevicePreviewActivity（被控端模拟屏）
~~~

- `MainActivity` 提供连接、灯光和计时控制；
- `RoomCoreService` 执行业务并保存模拟设备状态；
- `DevicePreviewActivity` 与 Service 一起运行在 `:room_core`；
- 被控端页面只展示最终状态，不直接执行控制命令。

## 2. 如何打开被控端页面

1. 在控制端连接服务；
2. 开灯、调节亮度或启动倒计时；
3. 点击“查看被控端效果”；
4. 观察明亮/暗黑模式和方框倒计时；
5. 点击“返回控制端”继续操作。

`AndroidManifest.xml` 中明确指定：

~~~xml
<activity
    android:name=".DevicePreviewActivity"
    android:exported="false"
    android:process=":room_core" />
~~~

因此这个页面不是主进程里的普通预览，而是真正运行在被控端进程。

## 3. 明亮和暗黑模式

被控端页面包含两个状态标签：

- **明亮模式**：`LightBinder` 回调 `enabled=true` 后选中；
- **暗黑模式**：`LightBinder` 回调 `enabled=false` 后选中。

页面中间的房间面板也会同步变化：

- 明亮模式使用暖黄色背景和发光圆形；
- 暗黑模式使用深色背景和熄灭圆形；
- 页面显示服务端确认的亮度百分比；
- Binder 不可用时显示等待状态。

效果更新发生在：

~~~java
onLightChanged(boolean enabled, int brightness, int servicePid)
~~~

而不是控制端按钮的点击事件。这保证展示的是被控端完成后的最终结果。

## 4. 方框式 XX / XX 倒计时

倒计时区域采用独立深色方框和等宽数字，例如：

~~~text
08 / 10
~~~

左侧是剩余秒数，右侧是本次倒计时总秒数。

为支持中途打开被控端页面，`ITimerCallback` 已扩展为：

~~~java
void onTimerChanged(
    int remainingSeconds,
    int totalSeconds,
    boolean running,
    int servicePid
);
~~~

如果只传剩余秒数，被控端页面在倒计时进行到一半时才打开，就无法知道原始总时长。现在 `TimerBinder` 保存并回调 `totalSeconds`，因此分母始终正确。

## 5. 页面为什么不提供控制按钮

被控端页面只负责展示，不提供“开灯”“关灯”“开始计时”等按钮，目的是保持职责分离：

~~~text
控制端：发送意图
被控端 Service：执行意图
被控端页面：展示执行后的状态
~~~

如果被控端页面也直接修改状态，就难以分辨效果到底来自 Binder 命令，还是页面自身操作。

## 6. 页面切换与 Service 生命周期

进入被控端页面时，它会在 `:room_core` 进程内绑定 `RoomCoreService` 并注册灯光、计时 Callback。返回控制端时，控制端会重新建立自己的 Binder 连接。

这样页面切换期间仍由绑定关系维持 Service，倒计时状态不会因为单纯打开预览页而主动停止。

## 7. 当前可验证的跨进程证据

运行时可以观察：

- 控制端 PID 与被控端 PID 不同；
- `DevicePreviewActivity` 和 `RoomCoreService` 的 PID 相同；
- 控制端先收到同步受理码；
- 灯光约 250ms 后才改变被控端页面状态；
- 倒计时由 `TimerBinder` 每秒推送；
- 主动断开和异常断开具有不同的重连语义。
