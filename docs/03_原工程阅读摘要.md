# 原工程阅读摘要

本示例不是凭题材猜出来的，而是基于当前 `BYD_SP_Annotated` 副本的源码、配置、文档、资源和接口表整理。

## 1. 内容盘点

- 文件总数：2602；
- 去除 `build/.gradle/.idea` 后的维护内容：1982 个文件；
- Java：443；AIDL：44；XML：340；PNG：1036；
- 可读取的源码/配置/文档文本：910 个、约 10.9 万行、读取错误 0；
- 另有 AAR、JAR、SO、签名文件和既有编译产物，按类型与来源模块盘点，没有把第三方二进制反编译结果当作架构依据。

## 2. 模块结论

```text
ComApp   可安装 APK：业务适配、UI、设置、埋点
Adapter  AIDL 合同、连接层、领域 Proxy/Stub
ToolLib  通用工具、平台配置、日志、Native Opus
Custom   Alexa Device Hosted Skill 参考实现
```

运行时核心是多 APK 协作和两级 Binder，而不是普通的单体 Activity 应用。

## 3. 接口表结论

6 份 `.xlsx` 已逐工作表、逐单元格读取：

- 两份 AppAdapter 表内容完全相同（SHA-256 相同），含 16 个领域工作表；
- 两份 UIAdapter 表内容完全相同（SHA-256 相同），含修改记录和 UI 两个工作表；
- 系统设置 PRD 含 5 个工作表，其中语音交互明细 114 行；
- 车控 PRD 含 5 个工作表，其中功能清单 93 行、语音交互明细 491 行。

接口表反复体现三个约束：

1. `zone`、`position`、`type/action` 不能混用；
2. 不同领域的成功码并不完全统一；
3. 大量操作以同步返回表示受理，以异步接口表示最终结果。

## 4. 抽取出的可迁移架构

```text
Service action / 显式组件
  → 一级 Router Binder
      → serviceName 路由
          → 二级领域 Binder
              → Proxy 封装远程调用
                  → Listener 异步回调
                      → Manager/UI 切回主线程
```

本示例保留这条链，删掉车机硬件、系统权限、车型配置和大量领域数量，使它能在普通 Android Studio 环境中运行。

## 5. 当前教学项目的展示扩展

在原工程抽取出的通信链上，当前示例增加了 `DevicePreviewActivity`。它与 `RoomCoreService` 同处 `:room_core` 进程，将灯光最终状态呈现为明亮/暗黑模式，并将计时回调呈现为 `XX / XX` 方框时钟。

这项扩展只增加状态可视化，没有改变“控制端 → Proxy → Binder → 服务端 → Callback”的主链路。
