# 单 APK 双进程与双 APK 对比

## 1. 两种方案

### 当前方案：单 APK 双进程

~~~text
ArchitectureLearningDemo.apk
├─ 主进程
│  └─ MainActivity（控制端）
└─ :room_core 进程
   ├─ RoomCoreService（被控端业务）
   └─ DevicePreviewActivity（被控端模拟屏）
~~~

### 拆分方案：两个独立 APK

~~~text
ControllerApp.apk
└─ MainActivity + Proxy
          │ 跨应用 Binder
          ▼
SmartRoomDevice.apk
└─ exported Service + DeviceActivity
~~~

## 2. 核心对比

| 维度 | 单 APK 双进程 | 两个独立 APK |
|---|---|---|
| Binder 是否真实 | 真实跨进程 Binder | 真实跨进程、跨应用 Binder |
| 进程隔离 | 有，主进程和 `:room_core` 独立 | 有，而且应用身份也独立 |
| 安装 | 安装一个 APK 即可 | 两个 APK 都要安装 |
| Android Studio 调试 | 一个工程、一套运行配置，最简单 | 需要两个 application 模块或两个工程 |
| Service 暴露 | `exported=false` 即可 | 通常必须 `exported=true` |
| 安全 | 同一应用 UID，权限配置简单 | 必须考虑自定义权限、签名校验和非法调用 |
| Service 定位 | 显式类名即可 | 需要跨包名显式 ComponentName 或稳定 Action |
| AIDL 版本 | 客户端和服务端总是随 APK 一起更新 | 两端可能版本不一致，需要兼容设计 |
| 发布升级 | 一次升级同时更新两端 | 两端可以独立升级，但必须管理兼容性 |
| 故障隔离 | 进程崩溃隔离，但仍属于同一应用包 | 应用、进程、数据和版本隔离更彻底 |
| 数据目录 | 同一应用沙箱，可共享应用级资源 | 各自独立沙箱，不能直接读取对方数据 |
| 签名 | 一个签名 | 可能需要双方签名一致或声明签名权限 |
| 学习成本 | 低，适合先理解 Binder 架构 | 高，更接近实际系统集成 |
| 自动化测试 | 安装一个 APK 后即可测试 | 测试前需要部署并协调两个 APK |
| 适用场景 | 教学、原型、同一产品内部多进程 | 独立团队、独立设备服务、可插拔客户端 |

## 3. 单 APK 双进程的优点

1. **学习路径短**

   一次编译、一次安装就能观察两个 PID、AIDL、回调和进程死亡。
2. **Binder 机制没有被省略**

   `android:process=":room_core"` 已经创建独立 Linux 进程，序列化、Binder 线程和 `RemoteException` 都是真实的。
3. **接口不会版本错配**

   AIDL 客户端和服务端由同一个 APK 编译，不会出现一端已经升级、另一端仍使用旧接口的问题。
4. **安全配置简单**

   Service 和被控端 Activity 都可以保持 `exported=false`，不会被其他应用随意调用。
5. **调试方便**

   一个 Android Studio 工程即可同时查看控制端、服务端和被控端页面代码。

## 4. 单 APK 双进程的缺点

1. **不能独立安装和升级**：控制端与被控端必须一起发布。
2. **应用身份没有分离**：虽然进程不同，但仍属于同一个 applicationId 和应用沙箱。
3. **无法完整演示跨应用安全**：不涉及 exported Service、自定义权限、签名校验和第三方客户端接入。
4. **产品边界不够真实**：如果设备服务由另一个团队或系统应用维护，单 APK 无法模拟独立版本节奏。

## 5. 两个 APK 的优点

1. **控制端和被控端完全独立**：可以拥有不同包名、签名、发布周期和数据目录。
2. **更贴近真实系统集成**：适合车机系统服务、智能家居设备服务或平台能力提供方。
3. **可支持多个控制端**：只要权限和接口允许，多个应用都可以绑定同一个设备 Service。
4. **升级更灵活**：被控端修复硬件逻辑时，不一定需要同时发布控制端。
5. **故障和职责边界更清晰**：可以分别统计崩溃、资源占用和版本信息。

## 6. 两个 APK 的缺点

1. **部署更复杂**：两个 APK 缺一不可，还要处理安装顺序和服务不可用状态。
2. **安全成本增加**：`exported=true` 后必须防止未授权应用绑定，通常需要 signature 权限或调用方校验。
3. **接口兼容更困难**：AIDL 方法一旦修改，就要考虑旧客户端、旧服务端和灰度升级。
4. **调试链更长**：需要同时查看两个进程、两个应用包的日志和版本。
5. **测试组合增加**：Controller v1/v2 与 Device v1/v2 会产生多种兼容组合。

## 7. 为什么本项目选择单 APK 双进程

本项目的目标是理解：

- 一级 Router Binder；
- 二级领域 Binder；
- Proxy 封装；
- 同步受理与异步完成；
- Binder 回调线程切换；
- 断线、超时和自动重连；
- 被控端最终状态展示。

这些机制在单 APK 双进程中都是真实存在。直接拆为两个 APK 不会让 Binder 本身“更真实”，主要增加的是应用边界、安全和版本治理。

~~~text
先用单 APK 双进程理解通信架构
→ 再拆两个 APK 学习安全与版本兼容
~~~

## 8. 从当前项目拆成两个 APK 需要改什么

1. 新建 `controller-app` 和 `device-app` 两个 application 模块；
2. 把 AIDL 合同抽到共享 library，或保证双方复制版本一致；
3. 将 `MainActivity`、Proxy、`RoomConnection` 放入控制端；
4. 将 `RoomCoreService`、`DevicePreviewActivity` 放入被控端；
5. 将 Service 改为可跨应用绑定；
6. 控制端使用包名和类名构造显式 `ComponentName`；
7. 增加 signature 级自定义权限；
8. 定义接口版本号和向后兼容策略；
9. 分别安装两个 APK 并进行组合测试。

## 9. 选择建议

- 目标是理解架构、做课堂演示：选择**单 APK 双进程**。
- 目标是模拟两个团队、两个系统应用：选择**两个 APK**。
- 目标是提供给第三方应用调用：选择**两个 APK**，并优先设计权限和版本兼容。
- 目标只是提高稳定性：先评估单 APK 多进程是否已经足够，不要仅为了复杂度拆包。
