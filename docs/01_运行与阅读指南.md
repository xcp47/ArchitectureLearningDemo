# 运行与阅读指南

## 1. 先运行，再看代码

推荐 API 35/36 的普通 Android 模拟器。项目没有车机私有 SDK、系统签名、特殊权限和第三方依赖。

运行后依次做六件事：

1. 看顶部 PID，确认 UI 与 Service 在两个进程；
2. 点击“开 / 关灯”，比较同步返回和异步回调的先后顺序；
3. 启动 10 秒倒计时，观察服务端每秒反向回调 UI；
4. 点击“查看被控端效果”，进入运行在 `:room_core` 的模拟屏；
5. 观察明亮/暗黑模式和方框式 `XX / XX` 倒计时；
6. 返回控制端后点击“断开”，确认两个二级 Binder 一起变为不可用。

## 2. 一次开灯调用经历了什么

```text
MainActivity
  └─ LightProxy.setEnabled(true)              本地 Java 调用
       └─ ILightService.setEnabled(true)       AIDL 二级 Binder
            └─ RoomCoreService.LightBinder    远端 :room_core 进程
                 ├─ return 0                  同步：请求已受理
                 └─ 250ms 后执行完成
                      └─ ILightCallback.onLightChanged(...)
                           └─ LightProxy 转到主线程
                                └─ MainActivity 更新界面
```

关键点：`MainActivity` 完全不知道 `IBinder` 和 `RemoteException`。跨进程细节被 `LightProxy` 隔离了。

## 3. 两级 Binder 是怎样拿到的

连接时只绑定一次 `RoomCoreService`：

```text
bindService(RoomCoreService)
  → IServiceRouter（一级 Binder）
       ├─ getService("LIGHT") → ILightService（二级 Binder）
       └─ getService("TIMER") → ITimerService（二级 Binder）
```

一级 Binder 不做具体业务，只做路由。这样再增加“窗帘”“音乐”等领域时，不需要再发布新的 Android Service。

## 4. 推荐阅读顺序

### 第一遍：只看合同

1. `ipc/ServiceNames.java`
2. `aidl/.../IServiceRouter.aidl`
3. `aidl/.../ILightService.aidl`
4. `aidl/.../ILightCallback.aidl`

先回答：系统有哪些能力？方法参数是什么？哪些结果同步返回，哪些异步通知？

### 第二遍：看服务端

阅读 `service/RoomCoreService.java`：

- `routerBinder` 对应总机；
- `LightBinder`、`TimerBinder` 对应分机；
- `RemoteCallbackList` 保存跨进程回调；
- Binder 线程只校验和排队，不做长耗时任务。

### 第三遍：看连接和代理

1. `client/RoomConnection.java`：只管理一级连接和状态机；
2. `client/BaseFeatureProxy.java`：模板方法骨架；
3. `client/LightProxy.java`：灯光领域填空；
4. `client/TimerProxy.java`：验证同一模板可以复用。

### 第四遍：最后看 UI

阅读 `MainActivity.java`。你会发现 UI 只调用普通 Java 风格的方法：

```java
lightProxy.setEnabled(true);
timerProxy.startTimer(10);
```

## 5. 状态机

```text
DISCONNECTED ── start() ──> CONNECTING ── onServiceConnected ──> CONNECTED
      ▲                         │                                  │
      │                         └─ 2 秒超时 ──┐                    │
      └──────── 500ms 快重试 ×5，之后 3s 慢重试 <── 远端断开 ─────┘
```

`stop()` 会设置“主动断开”语义，清除延时任务并禁止自动重连。这与原项目 `bManualDisconnect` 的作用相同。

## 6. 为什么回调要切主线程

AIDL 回调默认运行在 Binder 线程，不保证是 UI 主线程。`LightProxy` 和 `TimerProxy` 用主线程 `Handler` 再分发，`MainActivity` 才能安全更新 View。

这是一个重要边界：

```text
远端 Binder 线程 → Proxy 做线程切换 → UI 主线程
```

## 7. 常见问题

### Gradle 报 Unsupported class file major version 69

命令行正在用 JDK 25。把 Gradle JDK 改成 17 或 21。

### 页面只有一个 PID

先等灯光/计时器的首次异步回调。远端 PID 来自 `RoomCoreService`，回调前页面只知道 UI PID。

### 为什么不做成两个 APK

教学目标是理解真实 IPC，同时保持一键运行。`android:process=":room_core"` 已经创建独立 Linux 进程，Binder 行为是真实的；掌握后再拆成两个 APK，只需改 Service 暴露方式、包名和权限策略。

## 8. 被控端模拟屏怎么验证

`DevicePreviewActivity` 和 `RoomCoreService` 都声明在 `:room_core` 进程。进入页面后可以看到：

- 被控端页面 PID 与 Service 回调 PID 相同；
- 灯光只在 `onLightChanged` 回调后切换明亮/暗黑模式；
- 方框时钟显示“剩余秒数 / 总秒数”；
- `TimerBinder` 每秒推送一次计时状态；
- 页面只展示最终结果，不提供控制按钮。

详细说明见 [05_控制端与被控端说明.md](05_控制端与被控端说明.md)，单双 APK 取舍见 [06_单APK双进程与双APK对比.md](06_单APK双进程与双APK对比.md)。
